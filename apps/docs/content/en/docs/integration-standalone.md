# Standalone admin service

The recommended deployment model is to run the admin panel as **its own
service** — a small NestJS process that talks to the same database as
your main backend, but ships independently. The prebuilt React SPA is
served from the same process at `/admin`, so there is no separate
frontend to deploy.

```
┌────────────────┐   ┌──────────────────┐
│ Your main API  │   │ Modern Admin     │
│ (existing)     │   │ service (new)    │
│                │   │ NestJS + SPA     │
└────────┬───────┘   └─────────┬────────┘
         │                     │
         └──────────┬──────────┘
                    ▼
            ┌──────────────┐
            │  Postgres    │  (your app tables + ma_* tables)
            └──────────────┘
```

## Why standalone

- **Decoupled deploys.** Push admin changes without redeploying your
  main API; roll back the admin without affecting customers.
- **Smaller attack surface.** The admin service can sit behind a
  separate ingress (VPN-only, IP-allowlisted, etc.) while the main API
  stays public.
- **Independent scaling.** Admin traffic is bursty and human-paced;
  the main API may be 1000× busier. Scale them separately.
- **No framework lock-in.** Your main backend can be anything (Rails,
  Go, Django, …) — only the admin service has to be NestJS.
- **Shared truth.** Both services read/write the same Postgres tables.
  No sync layer, no event bus, no eventual consistency.

## Prerequisites

- **Bun** 1.3+ (used as both package manager and runtime).
- **PostgreSQL** reachable from the admin service.
- A **GitHub Packages read token** — Modern Admin is published to a
  private registry (`npm.pkg.github.com`). Create a PAT with the
  `read:packages` scope at https://github.com/settings/tokens.
- Optional: **Redis** for cross-instance cache invalidation when you
  run more than one admin replica.

## Scaffold the service

```sh
bun create @modern-admin admin-service
cd admin-service
```

This creates a complete NestJS project with:

- `prisma/schema.prisma` — your `Post` sample model plus the `ma_*`
  system tables (users, sessions, roles, logs, history, config, cache).
- `src/main.ts` — Nest bootstrap mounting Better Auth + the admin SPA.
- `src/admin.module.ts` — Modern Admin wired with Prisma + Better Auth.
- `src/resources/post.resource.ts` — sample `@AdminResource` to copy.
- `docker-compose.yml` — local Postgres + Redis for development.
- `.env.example` — required environment variables.

## Configure

```sh
cp .env.example .env
```

Edit `.env`:

| Variable              | Required | Description                                                                |
| --------------------- | -------- | -------------------------------------------------------------------------- |
| `DATABASE_URL`        | ✔        | Postgres URL — point at the **same database** as your main backend.        |
| `BETTER_AUTH_SECRET`  | ✔        | Random 32-byte hex. Generate with `bun run auth:secret`.                   |
| `BETTER_AUTH_URL`     |          | Public origin for auth cookies; defaults to `http://localhost:$PORT`.      |
| `MODERN_ADMIN_TOKEN`  | ✔ at install | Your GitHub PAT with `read:packages`. Read by `.npmrc` during install.     |
| `PORT` / `HOST`       |          | Default `3001` / `0.0.0.0`.                                                |
| `REDIS_URL`           |          | Enables cross-instance cache invalidation. Single-replica setups can skip. |
| `WEB_ORIGIN`          |          | CSV of allowed origins for CORS if you serve the SPA from a different host. |

## Apply migrations

The template's Prisma schema contains both your app's models (extend
freely) and the `ma_*` system tables Modern Admin needs. The
recommended workflow keeps the schemas **separate**:

1. Keep your main backend's schema as the source of truth for app
   tables. The admin service introspects them at startup via Prisma
   DMMF — no schema changes needed there.
2. Use the admin service's `prisma/schema.prisma` to **mirror** the app
   models (or `prisma db pull` from the shared database) plus the
   `ma_*` system tables.
3. Run `prisma migrate deploy` from one place only (typically your main
   backend's CI). Migrations for the `ma_*` tables ship as a single
   initial migration generated by `bun run db:migrate` once.

```sh
export MODERN_ADMIN_TOKEN=$(grep MODERN_ADMIN_TOKEN .env | cut -d= -f2)
bun install
bun run db:generate
bun run db:migrate   # only the very first time
bun run dev
```

The panel is now live at **http://localhost:3001/admin**.

## Seed the first admin user

The schema seeds zero users. Create one via a short script — Better
Auth hashes the password and Modern Admin reads the role:

```ts
// prisma/seed.ts
import { PrismaClient } from '@prisma/client'
import { uuidv7 } from '@modern-admin/core'

const prisma = new PrismaClient()
const password = await Bun.password.hash('admin12345', 'argon2id')

const user = await prisma.maUser.create({
  data: {
    id: uuidv7(),
    name: 'Admin',
    email: 'admin@example.com',
    emailVerified: true,
    role: 'admin',
  },
})
await prisma.maAccount.create({
  data: {
    id: uuidv7(),
    userId: user.id,
    providerId: 'credential',
    accountId: user.id,
    password,
  },
})
await prisma.maRole.upsert({
  where: { id: 'admin' },
  update: {},
  create: { id: 'admin', permissions: { '*': ['*'] }, isBuiltin: true },
})
```

Run with `bun run prisma/seed.ts`. Subsequent admin users can be added
through the **Users** resource inside the panel.

## Adding your own resources

For every database table you want to manage, create a sibling file
next to `post.resource.ts`:

```ts
// src/resources/product.resource.ts
import { Module } from '@nestjs/common'
import { AdminResource } from '@modern-admin/nest'
import type { PrismaResourceConfig } from '@modern-admin/adapter-prisma'
import { dmmf, prisma } from '../db.js'

const prismaSource = (modelName: string): (() => PrismaResourceConfig) => () => {
  const model = dmmf.datamodel.models.find((m) => m.name === modelName)
  if (!model) throw new Error(`Prisma model "${modelName}" not found`)
  const lowerFirst = (s: string): string =>
    s.length === 0 ? s : s.charAt(0).toLowerCase() + s.slice(1)
  return {
    model: model as never,
    client: prisma as never,
    clientKey: lowerFirst(modelName),
    enums: dmmf.datamodel.enums as never,
  }
}

@AdminResource({
  source: prismaSource('Product'),
  navigation: { icon: 'Package', group: 'Catalog' },
  listProperties: ['name', 'price', 'stock', 'updatedAt'],
})
export class ProductResource {}

@Module({ controllers: [ProductResource] })
export class ProductsAdminModule {}
```

Then import the new module in `src/admin.module.ts`:

```ts
import { ProductsAdminModule } from './resources/product.resource.js'

@Module({
  imports: [
    ModernAdminModule.forRoot({ /* … */ }),
    PostsAdminModule,
    ProductsAdminModule,   // ← added
  ],
})
export class AdminModule {}
```

See [Decorators](./decorators.md) for the full `@AdminResource` API
(properties, actions, hooks, navigation, sort, custom views).

## Sharing the database with your main backend

The admin service writes to the same Postgres database as your main
backend. A few conventions keep that safe:

- **Own your migrations.** Run all `prisma migrate deploy` invocations
  from a single CI job (typically the one for your main backend).
  Never `migrate dev` against a shared database from a developer
  laptop.
- **`ma_*` tables are namespaced.** All Modern Admin system tables use
  the `ma_` prefix so they cannot collide with your application
  tables. Treat them as owned by the admin service.
- **Use UUID v7 IDs for shared writes.** When the admin creates rows
  that the main backend will also touch, generate IDs with
  `uuidv7()` from `@modern-admin/core` — IDs are time-ordered and
  match the convention used by both sides.
- **Cache invalidation.** When the admin edits a row, the main backend
  must invalidate its own caches. Either subscribe to the realtime
  WebSocket events from `@modern-admin/realtime`, or invalidate on the
  next request via a short TTL. See [Cache](./cache.md).

## Deploying

The admin service is stateless and horizontally scalable. A typical
production setup:

- One or more replicas behind a load balancer (Fly.io, Render, Railway,
  Kubernetes `Deployment`, …).
- `REDIS_URL` set so all replicas see cache-invalidation events.
- `/admin` mounted behind your main domain via a path-prefix proxy
  (`mycompany.com/admin → admin-service`).
- Migrations run from CI (`prisma migrate deploy`) before traffic is
  cut over.
- Restrict ingress to the admin host — VPN, Cloudflare Access, or an
  allowlist of office IPs. Modern Admin enforces auth, but defence in
  depth is cheap.

```Dockerfile
# Dockerfile
FROM oven/bun:1.3-alpine
WORKDIR /app
COPY package.json bun.lock .npmrc ./
ARG MODERN_ADMIN_TOKEN
RUN MODERN_ADMIN_TOKEN=$MODERN_ADMIN_TOKEN bun install --production
COPY . .
RUN bun run db:generate
ENV NODE_ENV=production
EXPOSE 3001
CMD ["bun", "run", "start"]
```

Build with:

```sh
docker build --secret id=MODERN_ADMIN_TOKEN,src=.env.token \
  -t my-org/admin-service .
```

## Mounting under a subpath of your main API

When you want a single domain (`mycompany.com`) to host both your
existing API and the admin, use a reverse proxy in front:

```nginx
# nginx
location /admin {
  proxy_pass http://admin-service:3001;
  proxy_set_header Host $host;
  proxy_set_header X-Forwarded-Proto $scheme;
}

location / {
  proxy_pass http://main-api:8080;
}
```

No changes to either service are required — the admin already serves
its SPA + API under `/admin`.
