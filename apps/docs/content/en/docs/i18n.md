# Internationalisation (i18n)

Modern Admin ships a full i18n system out of the box. Every user-visible string in
the framework is localised — nothing is hardcoded in English.

The system has two complementary layers:

| Layer | What it covers |
|---|---|
| **`@modern-admin/i18n`** | UI strings: actions, pagination, toasts, validation messages, settings, charts, etc. |
| **`MetadataTranslations`** | Schema labels: resource names, property labels, enum option labels, navigation groups |

---

## Built-in locales

`@modern-admin/i18n` ships 9 locale bundles:

| Code | Language |
|---|---|
| `en` | English |
| `de` | Deutsch |
| `fr` | Français |
| `es` | Español |
| `it` | Italiano |
| `pt-BR` | Português (Brasil) |
| `ru` | Русский |
| `ja` | 日本語 |
| `pl` | Polski |

All locales cover the same set of keys. English (`en`) is always used as the
fallback when a key is missing in the active locale.

## The I18n class

`I18n` is a lightweight translation registry. It is intentionally minimal — you
can swap it for `i18next` or `react-intl` by wrapping the same interface.

```ts
import { I18n, builtinLocales } from '@modern-admin/i18n'

const i18n = new I18n({
  locales: builtinLocales,          // or a subset: [en, de, fr]
  defaultLocale: 'en',
  fallbackLocale: 'en',             // used when a key is absent in the active locale
})

i18n.t('common:save')               // → 'Save'
i18n.t('common:noRecordsHint', { resource: 'order' })
                                    // → 'Get started by creating a new order.'

i18n.setLocale('de')
i18n.t('common:save')               // → 'Speichern'

i18n.availableLocales()
// → [{ code: 'en', name: 'English' }, { code: 'de', name: 'Deutsch' }, …]
```

### I18nOptions

| Field | Type | Default | Description |
|---|---|---|---|
| `locales` | `LocaleBundle[]` | — | At least one locale required |
| `defaultLocale` | `string` | first locale | Code of the initially active locale |
| `fallbackLocale` | `string` | first locale | Used when a key is missing in the active locale |

### extend() — merge extra keys

`extend()` merges additional translations into an existing locale without rebuilding
the instance. Use this to override built-in strings or add keys for custom UI:

```ts
i18n.extend('en', {
  'common:appName': 'Acme Admin',
  'common:save': 'Save changes',
})

i18n.extend('de', {
  'common:appName': 'Acme Admin',
  'common:save': 'Änderungen speichern',
})
```

Later calls always win — there is no deep merge, keys are replaced atomically.

## React integration

### I18nProvider

Wrap your admin app with `I18nProvider`. It persists the selected locale to
`localStorage` under the key `modern-admin:locale` and restores it on next visit.

```tsx
import { AdminApp, I18nProvider } from '@modern-admin/react'
import { en, de, fr } from '@modern-admin/i18n'

function App() {
  return (
    <I18nProvider
      locales={[en, de, fr]}      // only ship the locales you need
      defaultLocale="en"
      fallbackLocale="en"
    >
      <AdminApp />
    </I18nProvider>
  )
}
```

`AdminApp` already wraps itself in `I18nProvider` with all 9 built-in locales when
you don't provide one explicitly. Pass your own `I18nProvider` to restrict the
language switcher to a subset or to inject custom translations.

### useI18n()

```tsx
import { useI18n } from '@modern-admin/react'

function MyComponent() {
  const { t, locale, setLocale, availableLocales } = useI18n()

  return (
    <button onClick={() => setLocale('de')}>
      {t('common:language')}: {locale}
    </button>
  )
}
```

| Member | Type | Description |
|---|---|---|
| `locale` | `string` | Currently active locale code |
| `setLocale(code)` | `(code: string) => void` | Switch locale + persist to localStorage |
| `t(key, params?)` | `(key, params?) => string` | Translate a key with optional interpolation |
| `availableLocales()` | `() => { code, name }[]` | All registered locales |
| `localizeResource(r)` | `(r: ResourceJSON) => ResourceJSON` | Apply MetadataTranslations to a resource |

`useI18n()` returns a no-op implementation when called outside a provider, so
components always render (keys are returned as-is).

### Template placeholders

Keys may contain `{placeholder}` tokens replaced at call time:

```ts
t('common:noRecordsHint', { resource: 'order' })
// → 'Get started by creating a new order.'

t('common:bulkDeleteConfirm', { count: 3 })
// → 'Delete 3 selected records?'

t('common:pageOf', { page: 2, total: 10 })
// → 'Page 2 of 10'
```

Placeholders whose value is `null` or `undefined` are left as `{placeholder}` in
the output.

## MetadataTranslations — schema label overrides

UI strings from `@modern-admin/i18n` cover the generic framework chrome. Resource
names, property labels, and enum values come from the ORM schema and can be
localised separately without modifying the database schema.

Pass a `metadataTranslations` map to `I18nProvider`:

```tsx
import type { MetadataTranslations } from '@modern-admin/react'

const translations: MetadataTranslations = {
  en: {
    navigation: {
      groups: {
        'Shop': 'Shop',
        'Users': 'Users',
      },
    },
    resources: {
      orders: {
        label: 'Orders',
        navigation: { name: 'Orders', group: 'Shop' },
        properties: {
          status: {
            label: 'Status',
            availableValues: {
              pending:   'Pending',
              shipped:   'Shipped',
              delivered: 'Delivered',
            },
          },
          'address.city': {
            label: 'City',
          },
        },
      },
    },
  },
  de: {
    navigation: {
      groups: {
        'Shop': 'Shop',
        'Users': 'Benutzer',
      },
    },
    resources: {
      orders: {
        label: 'Bestellungen',
        navigation: { name: 'Bestellungen', group: 'Shop' },
        properties: {
          status: {
            label: 'Status',
            availableValues: {
              pending:   'Ausstehend',
              shipped:   'Versendet',
              delivered: 'Geliefert',
            },
          },
          'address.city': {
            label: 'Stadt',
          },
        },
      },
    },
  },
}

function App() {
  return (
    <I18nProvider locales={[en, de]} metadataTranslations={translations}>
      <AdminApp />
    </I18nProvider>
  )
}
```

### Resolution order

For a given locale (e.g. `de`) and a resource property:

1. `metadataTranslations.de.resources.<resourceId>.properties.<path>.label`
2. `metadataTranslations.de.properties.<path>.label` (global property override)
3. `metadataTranslations.en.resources.<resourceId>.properties.<path>.label` (English fallback)
4. `metadataTranslations.en.properties.<path>.label`
5. Adapter-derived label from `PropertyDecorator.label()` (humanised path)

### MetadataTranslations shape

```ts
type MetadataTranslations = Record<
  string, // locale code
  {
    navigation?: {
      groups?: Record<string, string>   // group key → localised label
    }
    resources?: Record<string, {
      label?: string                    // resource display name
      name?: string                    // alias for label
      navigation?: {
        name?: string                   // sidebar label
        group?: string                  // sidebar group name
      }
      properties?: Record<string, {
        label?: string
        description?: string
        availableValues?: Record<string, string>  // enum value → label
        keyValueFields?: Record<string, {         // KeyValueEditor key → labels
          label?: string
          description?: string
          placeholder?: string
          availableValues?: Record<string, string>
        }>
      }>
    }>
    properties?: Record<string, {       // global property overrides (all resources)
      label?: string
      description?: string
      availableValues?: Record<string, string>
    }>
  }
>
```

## Customising built-in strings

### Override specific keys

Call `i18n.extend()` before rendering. With the React provider, build a custom
`LocaleBundle`:

```ts
import { en } from '@modern-admin/i18n'
import type { LocaleBundle } from '@modern-admin/i18n'

const myEn: LocaleBundle = {
  ...en,
  dict: {
    ...en.dict,
    'common:appName': 'Acme Admin',
    'auth:tagline': 'The admin panel for Acme Corp.',
  },
}
```

Pass `myEn` in place of (or alongside) the built-in `en` bundle.

### Replace the entire locale

To completely replace a locale (e.g. when migrating to `i18next`), build a
`LocaleBundle` object from your own translation files:

```ts
import enMessages from './translations/en.json'
import type { LocaleBundle } from '@modern-admin/i18n'

const en: LocaleBundle = {
  code: 'en',
  name: 'English',
  dict: enMessages,
}
```

The `dict` is a flat `Record<string, string>` — the same shape as the built-in
locale files.

## Adding a new locale

Create a `LocaleBundle` with your translations and pass it to `I18nProvider`:

```ts
// locales/ar.ts
import type { LocaleBundle } from '@modern-admin/i18n'
import { en } from '@modern-admin/i18n'

export const ar: LocaleBundle = {
  code: 'ar',
  name: 'العربية',
  dict: {
    ...en.dict,          // start from English, override what you've translated
    'common:appName': 'Modern Admin',
    'common:save': 'حفظ',
    'common:cancel': 'إلغاء',
    'common:delete': 'حذف',
    // … continue translating the rest of the keys
  },
}
```

```tsx
import { en, de } from '@modern-admin/i18n'
import { ar } from './locales/ar'

<I18nProvider locales={[en, de, ar]}>
  <AdminApp />
</I18nProvider>
```

The language switcher in the admin shell (`Settings → Language`) automatically
lists all registered locales.

**Required keys** — copy `en.ts` as your starting point. Missing keys fall back
to the `fallbackLocale` (default: `en`), so an incomplete locale degrades
gracefully rather than breaking the UI.

## Architecture notes

### i18n boundary

`packages/ui` components are **i18n-unaware**. They accept a `labels` prop with
English string defaults so they work standalone in tests and Storybook. The
`packages/react` layer is the translation boundary — it calls `t('ns:key')` and
passes the result via `labels` (or a named prop for single strings).

This means you can use `@modern-admin/ui` components in any app without
installing the i18n package.

### Locale persistence

The selected locale is stored in `localStorage` under `modern-admin:locale`. On
next page load `I18nProvider` reads this key and uses it as the initial locale,
falling back to `defaultLocale` if absent or if the stored code is not in the
registered locale list.

### Extending without React

If you use the `I18n` class directly (e.g. in a non-React context), you can call
`extend()` after construction to patch in custom strings without replacing the
whole bundle:

```ts
import { I18n, builtinLocales } from '@modern-admin/i18n'

const i18n = new I18n({ locales: builtinLocales, defaultLocale: 'en' })
i18n.extend('en', { 'common:appName': 'My Company Admin' })
```
